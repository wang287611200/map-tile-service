# 第3章 核心表设计详解

---

## regions 表：空间骨架

### 设计意图

`regions` 表是整个系统的**空间骨架**，定义了数据生产、下载、发布、存储的最小粒度单元。
每一行代表一个地理区域（国家、大区、省市），所有与数据相关的操作都以 `region_id` 为边界。

- 数据**生产**时：以 region 为单位切片、打包
- 数据**发布**时：以 region 为单位独立推进状态
- 数据**下载**时：以 region 为单位提供离线包
- 数据**存储**时：以 region 为单位组织 OSS 目录

### 层级结构示例

```
CN                         # 中国（根节点）
├── CN_NORTH               # 华北大区
│   ├── CN_BJ              # 北京市
│   ├── CN_TJ              # 天津市
│   └── CN_HEB             # 河北省
├── CN_EAST                # 华东大区
│   ├── CN_SH              # 上海市
│   ├── CN_JS              # 江苏省
│   └── CN_ZJ              # 浙江省
├── CN_SOUTH               # 华南大区
│   ├── CN_GD              # 广东省
│   └── CN_FJ              # 福建省
└── CN_WEST                # 西部大区
    ├── CN_SC              # 四川省
    └── CN_CQ              # 重庆市
```

## versions 表：时间主线

### 设计意图

`versions` 表是数据版本的**时间主线**，每行记录一个全局版本的元信息。
它描述的是"**这一份数据是什么**"，不涉及"哪些区域的数据已经就绪"。

### 记录什么

| 字段 | 含义 |
|------|------|
| `version_no` | 版本号（如 `v20260101`），体现数据时效性 |
| `format_version` | 该版本使用的数据格式版本，决定引擎兼容性 |
| `min_engine_version` | 需要的最低引擎版本 |
| `base_version_id` | 增量版本的基线，`NULL` 表示全量版本 |
| `status` | 全局状态（通常与各区域状态最高水位对齐） |

### 不记录什么

| 不在 versions 中存储的信息 | 原因 |
|---------------------------|------|
| 各区域是否已入库 | 由 `version_regions` 管理（区域粒度） |
| 数据文件路径 | 由 `version_regions.storage_path` 管理 |
| 瓦片文件哈希 | 由 `tile_index.etag` 管理 |
| 车辆当前使用版本 | 由 `vehicle_map_status` 管理 |

## version_regions 表：核心业务矩阵

### 设计意图

`version_regions` 是整个系统的**核心业务矩阵**，管理每个版本在每个区域的独立生命周期。
它是版本协商、灰度发布、回滚操作的核心依据。

### 关键特性

**区域独立发布**：同一版本 `v20260101`，不同区域可以处于不同发布阶段：

| version_id | region_id | status | 说明 |
|------------|-----------|--------|------|
| 1024 | CN_NORTH | PUBLISHED | 华北已全量 |
| 1024 | CN_EAST | CANARY | 华东灰度中 |
| 1024 | CN_SOUTH | DRAFT | 华南数据正在入库 |
| 1024 | CN_WEST | （无记录） | 西部尚未开始 |

这种设计使得：
- 数据生产进度不影响已完成区域的发布
- 某区域数据问题不影响其他区域
- 精细化控制发布节奏

## 三张表的协作全景图

```
┌─────────────────────────────────────────────────────────────────┐
│                         协作全景                                 │
│                                                                  │
│  regions (空间骨架)                                              │
│  ┌────────────────────────────────────────────────────────┐     │
│  │  CN  →  CN_NORTH  →  CN_BJ                             │     │
│  │         CN_EAST   →  CN_SH                             │     │
│  └────────────────────────────────────────────────────────┘     │
│              │ 区域粒度定义                                      │
│              ▼                                                   │
│  version_regions (业务矩阵)                                      │
│  ┌────────────────────────────────────────────────────────┐     │
│  │  v1024 × CN_NORTH → PUBLISHED  (tile_count, checksum)  │     │
│  │  v1024 × CN_EAST  → CANARY                             │     │
│  │  v1024 × CN_BJ    → PUBLISHED  (storage_path)          │     │
│  └────────────────────────────────────────────────────────┘     │
│              │ 归属到版本                                        │
│              ▼                                                   │
│  versions (时间主线)                                             │
│  ┌────────────────────────────────────────────────────────┐     │
│  │  v1024 → format_v4.0 → min_engine=3.0 → PUBLISHED      │     │
│  │  v1023 → format_v4.0 → min_engine=3.0 → DEPRECATED     │     │
│  │  v1022 → format_v3.5 → min_engine=2.5 → DEPRECATED     │     │
│  └────────────────────────────────────────────────────────┘     │
└─────────────────────────────────────────────────────────────────┘
```

## 三张表职责总结

| 表名 | 角色 | 谁写入 | 谁读取 | 变更频率 |
|------|------|--------|--------|----------|
| `regions` | 空间骨架（静态配置） | 运维/管理员 | 所有服务 | 极低（扩展区域时） |
| `versions` | 时间主线（版本元信息） | 版本管理服务 | 版本协商服务、API | 中（每次发版） |
| `version_regions` | 核心业务矩阵 | 数据入库服务、发布服务 | 版本协商、灰度服务、车端 API | 高（每次区域状态变更） |
