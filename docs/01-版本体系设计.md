# 第1章 版本体系设计

---

## 4.1 三维版本空间

地图数据版本管理涉及三个相互独立的维度，形成一个三维版本空间：

```
        引擎版本 (能力)
            │
         4.0│         ✦ 最新引擎
         3.5│       ✦
         3.0│     ✦
         2.5│   ✦
            └──────────────────── 数据格式版本 (兼容性)
           /   3.0  3.5  4.0  4.1
          /
    数据内容版本 (时效性)
   20240101
   20240601
   20250101
   20260101
```

| 维度 | 含义 | 变更频率 |
|------|------|----------|
| **数据内容版本** | 地图采集数据的时效性（道路、POI 等实际内容） | 高频（每月/每季度） |
| **数据格式版本** | 瓦片编码协议、字段结构的版本（影响引擎兼容性） | 低频（每年 1-2 次） |
| **引擎版本** | 车载地图渲染引擎的版本（解析能力） | 中频（随 OTA 更新） |

---

## 4.2 数据格式版本 vs 数据内容版本

| 对比维度 | 数据格式版本 (`format_version`) | 数据内容版本 (`version_no`) |
|----------|--------------------------------|-----------------------------|
| **本质** | 编码协议版本（如 PBF schema） | 采集日期 + 内容快照 |
| **变更原因** | 新增字段、改变编码方式、重构数据结构 | 道路更新、POI 变化、新建地物 |
| **变更频率** | 低频（每年 1-2 次）| 高频（每月/每季度） |
| **影响范围** | 需要引擎升级才能解析新格式 | 引擎无感知，仅内容变化 |
| **示例** | `format_v3.0` → `format_v4.0` | `v20240101` → `v20260101` |
| **兼容性约束** | 强约束：引擎必须支持该格式版本 | 无约束：引擎与内容解耦 |

## 4.3 版本生命周期状态机

```
                    ┌──────────┐
              创建  │  DRAFT   │  内容入库中，尚未就绪
                    └────┬─────┘
                         │ 质检通过
                         ▼
                    ┌──────────┐
                    │ TESTING  │  质检阶段，仅测试车辆可用
                    └────┬─────┘
                         │ 开始灰度
                         ▼
                    ┌──────────┐
                    │  CANARY  │  灰度发布，部分车辆可用
                    └────┬─────┘
                         │ 全量发布
                         ▼
                    ┌──────────┐
              ┌──── │PUBLISHED │ ────────────────┐
              │     └────┬─────┘                 │
              │          │ 新版本全量后废弃        │ 发现严重问题
              │          ▼                        ▼
              │     ┌──────────────┐      ┌─────────────┐
              │     │  DEPRECATED  │      │ ROLLED_BACK │
              │     └──────────────┘      └─────────────┘
              │
              └── 任意阶段均可触发 ROLLED_BACK（回滚）
```

| 状态 | 说明 | 可见性 |
|------|------|--------|
| `DRAFT` | 版本创建，区域数据正在写入 | 仅内部系统 |
| `TESTING` | 质检通过，进入测试阶段 | 测试车辆 |
| `CANARY` | 灰度发布，按 vehicle_id 分桶 | 灰度比例内的车辆 |
| `PUBLISHED` | 全量发布，所有兼容车辆可用 | 所有车辆 |
| `DEPRECATED` | 被更新版本取代，逐步下线 | 仍在使用的旧版车辆（降级） |
| `ROLLED_BACK` | 因问题主动回滚，不再提供服务 | 不可见 |

### 版本保留策略

| 状态 | 保留数量/时间 | 说明 |
|------|--------------|------|
| `PUBLISHED` | 保留最近 **3** 个 | 保障回滚窗口 |
| `DEPRECATED` | 保留 **30 天** | 确保仍在使用旧版本的车辆完成迁移 |
| `ROLLED_BACK` | 保留 **7 天** | 保留用于问题排查，之后清理 |

## 4.4 版本标识策略

| 策略类型 | 格式示例 | 使用场景 |
|----------|----------|----------|
| **日期版本** | `v20260101` | 数据内容版本（体现时效性） |
| **语义化版本** | `v3.5.2` | 引擎版本、格式版本（体现兼容性） |
| **三段式版本** | `v20260101-fmt4.0-eng3.5` | 版本协商时的完整标识（含格式和引擎约束） |
| **内容哈希** | `sha256:a3f9...` | 瓦片文件唯一标识（用于 ETag 和完整性校验） |

> **设计原则**：版本标识应在 URL 路径中编码，而非通过查询参数传递，以支持 CDN 静态缓存。
