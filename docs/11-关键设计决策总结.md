# 第11章 关键设计决策总结

---

## 设计决策对比汇总表

| 设计点 | 采用方案 | 被放弃的方案 | 采用原因 |
|--------|----------|-------------|----------|
| **存储效率** | Copy-on-Write（只存变化瓦片） | 每版本全量存储 | 节省 ~95% 存储空间 |
| **下载效率** | CDN + 增量包 + Range 断点续传 | 全量下载 | 流量节省 50-90% |
| **灰度发布** | vehicle_id 哈希分桶 + 兼容性准入双重过滤 | 随机选择 / IP 地域 | 同一车辆灰度结果稳定，且保证兼容性 |
| **版本回滚** | 数据不可变，回滚只切换状态字段 | 删除新版本数据 | 秒级完成，无数据丢失风险 |
| **数据一致性** | SHA256 校验和（入库 + 下载双重校验） | CRC32 / 不校验 | SHA256 防碰撞，适合安全场景 |
| **在线/离线协同** | 在线优先 + 离线包兜底 | 纯离线 / 纯在线 | 兼顾数据新鲜度和弱网可用性 |
| **存储清理** | 版本过期自动清理（DEPRECATED 30天后删除） | 手动清理 | 自动化，避免存储无限增长 |
| **CDN 架构** | 双链路（动态版本协商 + 静态瓦片直连 CDN） | 单链路（所有请求经过业务服务） | 静态瓦片无需业务逻辑，CDN 直连 OSS 降低延迟 |
| **缓存新鲜度** | 版本化 URL + 版本协商驱动更新 | URL 固定 + 主动 CDN 失效 | 版本化 URL 天然不可变，无需主动清缓存 |
| **引擎兼容** | `compatibility_matrix` 规则表 + 服务端决策 | 客户端自行判断 / 硬编码规则 | 规则集中管理，易更新；服务端决策避免旧客户端不升级的问题 |

## 三层防线总结

版本更新感知能力由三层机制共同保障：

```
┌───────────────────────────────────────────────────────────────────────┐
│                          三层防线                                      │
│                                                                       │
│  第一层：MQTT 推送                                                     │
│  ┌──────────────────────────────────────────────────────────────┐    │
│  │  触发时机：服务端发布新版本后立即推送                           │    │
│  │  感知延迟：秒级（< 3s）                                        │    │
│  │  流量消耗：极低（仅推送 JSON 消息，< 1KB）                     │    │
│  │  可靠性：中等（依赖 MQTT 连接，弱网或断连时可能丢失）           │    │
│  └──────────────────────────────────────────────────────────────┘    │
│                                                                       │
│  第二层：版本清单轮询                                                  │
│  ┌──────────────────────────────────────────────────────────────┐    │
│  │  触发时机：自适应定时（5分钟 ~ 2小时，按车辆状态调整）          │    │
│  │  感知延迟：分钟级（5~120分钟）                                 │    │
│  │  流量消耗：低（304 无变化时零字节，有变化时 < 10KB）           │    │
│  │  可靠性：高（HTTP 轮询，可重试，对弱网友好）                   │    │
│  └──────────────────────────────────────────────────────────────┘    │
│                                                                       │
│  第三层：版本化 URL + 版本协商                                         │
│  ┌──────────────────────────────────────────────────────────────┐    │
│  │  触发时机：车端每次请求瓦片时（实时）                          │    │
│  │  感知延迟：最终一致性（版本协商后立即生效）                     │    │
│  │  流量消耗：正常请求的一部分                                    │    │
│  │  可靠性：最高（每次请求都隐式做版本检查，无法跳过）             │    │
│  └──────────────────────────────────────────────────────────────┘    │
└───────────────────────────────────────────────────────────────────────┘
```

### 三层防线对比表

| 防线 | 感知延迟 | 流量消耗 | 可靠性 | 场景 |
|------|----------|----------|--------|------|
| **MQTT 推送** | 秒级（< 3s） | 极低 | 中（弱网可丢失） | 有变化时主动通知，最快感知 |
| **版本清单轮询** | 分钟级（5~120 min） | 低 | 高（HTTP 可重试） | 推送丢失后的兜底保障 |
| **版本化 URL** | 最终一致性 | 正常 | 最高（每次请求隐式检查） | 最终防线，确保不使用错误版本 |

## 核心架构决策原则

1. **数据不可变**：OSS 中的瓦片文件一旦写入，永不修改，只新增。版本管理通过状态字段实现，而非数据覆盖。

2. **服务端决策**：所有涉及兼容性判断、灰度准入的逻辑均在服务端执行，客户端无需感知规则细节，确保规则可动态更新。

3. **版本化 URL**：将"缓存失效"问题转化为"版本发现"问题，CDN 缓存永不失效，只需通过版本协商发现新版本号。

4. **区域独立发布**：不同地理区域的数据版本独立管理，一个区域的问题不影响其他区域，实现精细化发布控制。

5. **降级优先**：在任何故障场景下，优先保障导航可用性（使用旧版本离线数据），而非强制要求最新数据。
