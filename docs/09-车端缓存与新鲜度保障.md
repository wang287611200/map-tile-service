# 第9章 车端缓存与新鲜度保障

---

## 11.1 核心矛盾

| 缓存的价值 | 新鲜度的要求 |
|-----------|-------------|
| ✅ 降低流量消耗（车端带宽昂贵） | ⚠️ 道路变化（新建/改建/封路）需及时生效 |
| ✅ 降低请求延迟（离线场景） | ⚠️ POI 数据时效性（营业时间/地址变更） |
| ✅ 支持离线导航（隧道/地下停车场） | ⚠️ 安全相关信息（限速标志/施工区域） |

**解决思路**：通过版本驱动的缓存更新机制，将"缓存失效"问题转化为"版本发现"问题，
同时通过推送+轮询双通道确保版本变化能及时到达车端。

---

## 11.2 版本驱动的缓存更新机制

```
版本协商 → 发现新版本号 → URL 发生变化 → 自动拉取新数据 → 旧缓存自然淘汰
```

### 5个核心优势

- ✅ **无需主动清除缓存**：旧版本 URL 的缓存自然过期（30天），无需推送 CDN 清除指令
- ✅ **无缓存污染风险**：版本化 URL 保证同一 URL 的内容永远一致
- ✅ **支持离线降级**：旧版本缓存在新版本下载完成前仍然可用
- ✅ **增量更新**：只有有变化的瓦片才有新 URL，变化率 ~5%，95% 的请求命中旧缓存
- ✅ **CDN 友好**：版本化 URL 可使用 `immutable` 标志，CDN 无限期缓存

## 11.3 推送+轮询双通道

### MQTT 推送（主动通知）

服务端在新版本发布后，通过 MQTT 向受影响的车辆推送变更通知：

```json
{
  "type": "MAP_VERSION_UPDATED",
  "version_id": 1024,
  "version_no": "v20260101",
  "affected_regions": ["CN_BJ", "CN_TJ"],
  "affected_tile_ranges": [
    {"region": "CN_BJ", "zoom_min": 10, "zoom_max": 17, "bounds": {...}},
    {"region": "CN_TJ", "zoom_min": 12, "zoom_max": 17, "bounds": {...}}
  ],
  "change_summary": {
    "road_changes": 1230,
    "poi_changes": 4500,
    "priority": "normal"
  },
  "published_at": "2026-03-10T08:00:00Z"
}
```

### 车端收到推送后的处理流程

```python
def on_map_update_push(message: dict):
    # Step 1: 解析受影响的区域和瓦片范围
    affected_regions = message['affected_regions']
    affected_ranges = message['affected_tile_ranges']

    # Step 2: 在本地缓存中将受影响的瓦片标记为 STALE
    for region in affected_regions:
        tile_range = get_range_for_region(region, affected_ranges)
        db.execute("""
            UPDATE cached_tiles
            SET cache_status = 'STALE'
            WHERE region_id = :region
              AND zoom_level BETWEEN :zoom_min AND :zoom_max
              AND cache_status = 'FRESH'
        """, region=region, zoom_min=tile_range.zoom_min, zoom_max=tile_range.zoom_max)

    # Step 3: 按优先级调度预拉取
    priority_regions = get_priority_regions(affected_regions)
    for region in priority_regions:
        schedule_prefetch(region, priority=message['change_summary']['priority'])
```

### 版本清单轮询（兜底保障）

| 车辆状态 | 轮询间隔 | 说明 |
|----------|----------|------|
| 行驶中 + 导航激活 | **5 分钟** | 最高频率，导航对数据时效性要求最高 |
| 行驶中 + 未导航 | **15 分钟** | 次高频率 |
| 驻车 + WiFi 连接 | **30 分钟** | 可利用 WiFi 预下载 |
| 驻车 + 蜂窝网络 | **2 小时** | 节省流量 |
| 熄火 | **不轮询** | 避免后台消耗电量 |

### 轮询比对流程

```
车端                                    版本清单服务
  │                                          │
  │ GET /api/v1/manifest                     │
  │ If-None-Match: "sha256:prev_hash..."     │
  │─────────────────────────────────────────>│
  │                                          │
  │    ← 304 Not Modified（无变化）           │
  │<─────────────────────────────────────────│
  │    无需操作                              │
  │                                          │
  │    ← 200 OK（有变化）                    │
  │    {region_versions: {...},              │
  │     manifest_hash: "new_hash..."}        │
  │<─────────────────────────────────────────│
  │                                          │
  │ 逐区域比对 manifest_hash                 │
  │ ├── CN_BJ 版本号变化 → 标记 STALE        │
  │ ├── CN_TJ 版本号不变 → 无需操作          │
  │ └── CN_NORTH 版本号变化 → 调度预拉取     │
```

## 11.4 Merkle Tree 加速比对

对于大规模瓦片变化检测，使用 Merkle Tree 结构加速比对，避免逐瓦片检查。

### 4层树结构

```
根节点（全国）
    └── 大区节点（CN_NORTH / CN_EAST / CN_SOUTH / CN_WEST）
            └── 省市节点（CN_BJ / CN_TJ / CN_SH ...）
                    └── Zoom 节点（Zoom 10 / 11 / 12 ... 17）
                            └── 瓦片叶子节点（tile_hash）
```

### 类似 Git Tree Diff 的比对过程

```python
def merkle_diff(old_root_hash: str, new_root_hash: str) -> list[str]:
    """
    返回发生变化的区域列表，类似 git diff 的树形差异比对
    """
    if old_root_hash == new_root_hash:
        return []  # 根节点相同，全国无变化

    changed_regions = []
    for region in ['CN_NORTH', 'CN_EAST', 'CN_SOUTH', 'CN_WEST']:
        old_region_hash = get_region_hash(old_root_hash, region)
        new_region_hash = get_region_hash(new_root_hash, region)

        if old_region_hash != new_region_hash:
            # 该大区有变化，继续向下细化
            changed_cities = diff_region(old_region_hash, new_region_hash, region)
            changed_regions.extend(changed_cities)

    return changed_regions
```

## 11.5 车端缓存数据结构

### 本地目录结构

```
{cache_root}/
├── meta/
│   ├── manifest.json                  # 最新版本清单
│   ├── vehicle_config.json            # 车辆配置（引擎版本、订阅区域）
│   └── sync_state.json                # 同步状态（上次轮询时间等）
├── index/
│   └── tile_cache.db                  # SQLite 缓存索引（见下方）
└── tiles/
    ├── road_network/
    │   ├── v20250601/                 # 旧版本（降级备用）
    │   │   └── 14/13708/6411.pbf
    │   └── v20260101/                 # 当前版本
    │       └── 14/13708/6411.pbf
    └── offline/
        └── CN_NORTH/
            └── v20260101/
                └── full.pkg
```

### SQLite 缓存索引表

```sql
CREATE TABLE cached_tiles (
    tile_key        TEXT PRIMARY KEY,          -- "{data_type}/{version}/{z}/{x}/{y}"
    data_type       TEXT NOT NULL,
    zoom_level      INTEGER NOT NULL,
    tile_x          INTEGER NOT NULL,
    tile_y          INTEGER NOT NULL,
    region_id       TEXT NOT NULL,
    content_version TEXT NOT NULL,             -- 版本号，如 v20260101
    etag            TEXT,                      -- 内容哈希，用于一致性校验
    cache_status    TEXT NOT NULL              -- FRESH / STALE / EXPIRED / DEGRADED
                    CHECK (cache_status IN ('FRESH', 'STALE', 'EXPIRED', 'DEGRADED')),
    cached_at       INTEGER NOT NULL,          -- Unix timestamp
    ttl_seconds     INTEGER NOT NULL DEFAULT 2592000,  -- 30天
    access_count    INTEGER NOT NULL DEFAULT 0,
    last_accessed   INTEGER,                   -- 用于 LRU 淘汰
    file_path       TEXT NOT NULL,             -- 本地文件路径
    file_size       INTEGER NOT NULL
);

-- 索引：按区域快速查询
CREATE INDEX idx_region ON cached_tiles(region_id, zoom_level);
-- 索引：按版本快速查询（用于版本升级后批量标记 STALE）
CREATE INDEX idx_version ON cached_tiles(content_version);
-- 索引：LRU 淘汰（按最近访问时间排序）
CREATE INDEX idx_lru ON cached_tiles(last_accessed ASC);
-- 索引：TTL 过期（按过期时间排序）
CREATE INDEX idx_ttl ON cached_tiles(cached_at + ttl_seconds ASC);
```

## 11.6 缓存淘汰策略

### 4级优先级（从高到低淘汰）

| 优先级 | 淘汰目标 | 策略 |
|--------|----------|------|
| **优先级1（最先淘汰）** | `STALE` 状态 + 非当前可视区域的瓦片 | 已过期且不在使用，优先清理 |
| **优先级2** | `EXPIRED` 状态 + 低访问频次（`access_count < 3`） | 冷数据，优先清理 |
| **优先级3** | 低 Zoom 级别（zoom ≤ 6）的旧版本瓦片 | 低缩放级别数据量小，易重新下载 |
| **优先级4（最后淘汰）** | 远距离区域（距当前位置 > 200km）的瓦片 | 短期内不会使用 |

### 永不淘汰

- 当前版本的**离线包**文件（`offline/*.pkg`）不参与 LRU 淘汰

### 三级水位线

| 水位线 | 触发条件 | 处理动作 |
|--------|----------|----------|
| **警告水位（90%）** | 磁盘占用 > 90% | 开始按优先级淘汰第1、2级 |
| **告警水位（95%）** | 磁盘占用 > 95% | 加速淘汰，暂停新的预拉取 |
| **紧急水位（99%）** | 磁盘占用 > 99% | 强制淘汰，包括当前版本的旧区域瓦片 |

## 11.7 智能预拉取策略

### 行驶方向预拉取（前方 5km）

```
当前位置 ●
          ↑ 行驶方向
    ┌─────────────────┐
    │                 │
    │  预拉取区域      │
    │  前方 5km        │
    │  当前 zoom±2    │
    │                 │
    └─────────────────┘
          ↑ 已拉取区域
          ●（当前位置）
```

```python
def prefetch_driving_direction(current_pos, speed, heading):
    # 计算前方5km的边界框
    bbox = calc_bbox_ahead(current_pos, distance=5000, heading=heading)
    current_zoom = speed_to_zoom(speed)  # 速度越快，预拉取越低的zoom（更大范围）

    prefetch_tiles(bbox, zoom_range=(current_zoom - 2, current_zoom + 1))
```

### 导航路线预拉取（20km）

当导航激活时，预拉取整条路线范围内的瓦片（最远 20km）：

```python
def prefetch_navigation_route(route: List[Point]):
    # 对路线的每个关键节点，预拉取周边范围
    for point in route[:50]:  # 只预拉取前50个节点（约20km）
        bbox = point.to_bbox(radius=500)  # 节点周边500m
        prefetch_tiles(bbox, zoom_range=(14, 17))
```

### 高频区域预拉取

统计车辆历史高频访问区域（家庭/公司周边），在 WiFi 下预拉取最新版本：

```python
def prefetch_frequent_regions():
    frequent_regions = get_top_frequent_regions(top_n=5)
    for region in frequent_regions:
        latest_version = get_latest_version(region.id)
        if region.cached_version != latest_version:
            schedule_prefetch(region.id, priority='LOW')
```

### WiFi 批量预拉取

接入 WiFi 时，利用高带宽批量更新：

```python
def on_wifi_connected():
    # 获取所有 STALE 瓦片列表
    stale_tiles = db.query("SELECT * FROM cached_tiles WHERE cache_status = 'STALE'")

    # 按 region + zoom 分批预拉取，避免阻塞正常请求
    batch_prefetch(stale_tiles, concurrency=8, rate_limit='50Mbps')
```
