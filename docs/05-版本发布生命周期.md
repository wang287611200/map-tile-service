## 7.2 阶段1：版本创建

版本管理服务在数据库中创建版本记录，此时 `version_regions` 表为空。

```sql
-- 创建版本记录
INSERT INTO versions (
    data_type, version_no, format_version,
    min_engine_version, base_version_id, status
) VALUES (
    'road_network',
    'v20260101',
    'format_v4.0',
    '3.0.0',
    1023,        -- 上一个版本作为增量基线
    'DRAFT'
)
RETURNING version_id;
-- 返回 version_id = 1024

-- 记录审计日志
INSERT INTO version_publish_log (version_id, action, to_status, operator)
VALUES (1024, 'CREATE', 'DRAFT', 'data-pipeline-bot');
```

此时 `version_regions` 表中 **没有** `version_id=1024` 的记录，表示各区域数据尚未开始入库。

## 7.4 阶段3：质量验证

质检系统对各区域数据逐项验证，全部通过后更新状态。

### 质检项

| 质检类型 | 检查内容 | 通过标准 |
|----------|----------|----------|
| **完整性验证** | SHA256 校验和、瓦片数量比对 | 与入库记录完全一致 |
| **拓扑验证** | 路网连通性、无孤立路段 | 连通率 > 99.9% |
| **渲染验证** | 抽样瓦片渲染效果截图 | 无黑屏/花屏/文字乱码 |
| **回归验证** | 与上一版本的关键 POI 对比 | 重要地物不丢失 |

### 质检通过后更新状态

```sql
-- 华北区域质检通过
UPDATE version_regions
SET status = 'TESTING', updated_at = NOW()
WHERE version_id = 1024 AND region_id = 'CN_NORTH';

INSERT INTO version_publish_log (version_id, region_id, action, from_status, to_status, operator)
VALUES (1024, 'CN_NORTH', 'QUALITY_PASS', 'DRAFT', 'TESTING', 'qa-system');
```

## 7.5 阶段4：灰度发布

将特定区域的状态更新为 `CANARY`，并配置灰度比例：

```sql
UPDATE version_regions
SET status = 'CANARY', updated_at = NOW()
WHERE version_id = 1024 AND region_id = 'CN_NORTH';

-- 设置灰度比例（存储在单独的配置表中）
INSERT INTO canary_config (version_id, region_id, ratio, started_at)
VALUES (1024, 'CN_NORTH', 20, NOW())  -- 20% 的车辆进入灰度
ON CONFLICT (version_id, region_id) DO UPDATE SET ratio = 20;
```

### 灰度判断逻辑（哈希分桶）

```python
def is_in_canary(vehicle_id: str, version_id: int, region_id: str) -> bool:
    config = db.get_canary_config(version_id, region_id)
    if not config:
        return False
    # 使用 vehicle_id 的哈希值分桶，确保同一辆车始终稳定进入/不进入灰度
    bucket = int(hashlib.sha256(vehicle_id.encode()).hexdigest(), 16) % 100
    return bucket < config.ratio
```

### 灰度监控指标

| 指标 | 正常范围 | 异常阈值 | 处理动作 |
|------|----------|----------|----------|
| 瓦片请求成功率 | > 99.5% | < 99% | 自动暂停灰度 |
| 渲染崩溃率 | < 0.01% | > 0.1% | 触发告警+人工介入 |
| 平均下载时延 | < 200ms | > 500ms | 触发告警 |
| 用户投诉率 | 基准值 | 基准值+10% | 触发告警 |

### 版本协商中的灰度处理

```python
# 在版本协商逻辑中，CANARY 状态的版本只对分桶内的车辆可见
for candidate in sorted_candidates:
    if candidate.vr_status == 'PUBLISHED':
        return candidate  # 所有车辆可用
    elif candidate.vr_status == 'CANARY':
        if is_in_canary(vehicle_id, candidate.version_id, region_id):
            return candidate  # 仅灰度车辆可用
        # 否则继续查找下一个 PUBLISHED 版本
```

### 废弃旧版本

```sql
-- 检查全局状态：当 PUBLISHED 版本数量 > 3，废弃最旧的
UPDATE version_regions
SET status = 'DEPRECATED', deprecated_at = NOW()
WHERE version_id = (
    SELECT version_id FROM versions
    WHERE data_type = 'road_network' AND status = 'PUBLISHED'
    ORDER BY version_no ASC
    LIMIT 1
);
```

## 7.7 版本回滚

当新发布的版本出现严重问题时，通过修改状态实现**秒级回滚**：

```sql
-- 将问题版本在所有区域标记为 ROLLED_BACK
UPDATE version_regions
SET status = 'ROLLED_BACK', updated_at = NOW()
WHERE version_id = 1024;

-- 写入审计日志
INSERT INTO version_publish_log (version_id, action, from_status, to_status, operator, reason)
VALUES (1024, 'ROLLBACK', 'PUBLISHED', 'ROLLED_BACK', 'oncall-engineer',
        '华北区域出现路网显示错误，影响导航');
```

### 推送回滚通知

```json
{
  "type": "VERSION_ROLLED_BACK",
  "version_id": 1024,
  "reason": "路网显示异常",
  "fallback_version": "v20250601"
}
```

回滚后，**版本协商服务自动降级**：由于 `ROLLED_BACK` 版本不在可用状态列表中，
版本协商查询会自动返回上一个 `PUBLISHED` 版本（`v20250601`）。

### 回滚的三大优势

| 优势 | 说明 |
|------|------|
| **OSS 旧数据仍在** | 数据文件从未删除，仅状态字段变更 |
| **只改状态字段** | 回滚操作仅执行 `UPDATE version_regions SET status='ROLLED_BACK'` |
| **秒级完成** | 无需重新部署、无需重新上传数据，状态变更即时生效 |
